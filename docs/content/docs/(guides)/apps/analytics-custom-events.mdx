Use Stack Auth custom analytics events when you want application-specific product, billing, support, or operational events to land in the same Analytics dataset as Stack-managed events.

When replay recording is enabled in the browser SDK, client-side `trackEvent()` calls automatically attach the current replay linkage once a replay has been established. That linkage includes `browserSessionId`, `sessionReplayId`, and `sessionReplaySegmentId`, and you can pass any of them explicitly in `trackEvent(..., options)` on both the client and server SDKs.

## Recipes

### Client-side (all frameworks)

Call `trackEvent()` directly on your `StackClientApp` instance:

```ts
app.trackEvent("upgrade.clicked", {
  plan: "pro",
  source: "pricing-page",
});
```

### Server-side (all frameworks)

Call `trackEvent()` on your `StackServerApp` instance, passing the request for user attribution:

```ts
serverApp.trackEvent("support.ticket.created", {
  channel: "chat",
}, req);
```

Any Fetch API `Request`, Express-style `req`, or Hono `c.req.raw` works as the 3rd argument.

Server `trackEvent()` is non-blocking — it buffers events and flushes them in the background. Call `await serverApp.flushAnalytics()` if you need to ensure all events are sent before the process exits.

### Serverless environments

In serverless runtimes (Vercel, Cloudflare Workers, Lambda), the function may shut down before buffered analytics are flushed. Pass `waitUntil` to the constructor to extend the function's lifetime:

```ts
// Vercel
import { waitUntil } from "@vercel/functions";

const serverApp = new StackServerApp({
  projectId: "...",
  secretServerKey: "...",
  waitUntil,
});
```

```ts
// Cloudflare Workers
export default {
  async fetch(request, env, ctx) {
    const serverApp = new StackServerApp({
      projectId: "...",
      secretServerKey: "...",
      waitUntil: (p) => ctx.waitUntil(p),
    });
    // ...
  },
};
```

If omitted, the SDK auto-detects `globalThis.waitUntil` (available in some runtimes). Falls back to fire-and-forget if unavailable.

### Error capture

Errors are auto-captured in the browser via `window.onerror` and `unhandledrejection`. For manual error capture:

```ts
// Client
app.captureException(err);
app.captureException(err, { component: "checkout" });

// Server — pass request for user attribution
serverApp.captureException(err, req);
serverApp.captureException(err, req, { route: "/api/checkout" });
```

Inside a span, you can also use `span.recordException(err)` which sets the span status to error and attaches the error metadata.

## Naming Rules

- Reserve the `$` prefix for Stack-managed events
- Use only letters, numbers, `.`, `_`, `:`, and `-`
- Keep payloads as JSON-serializable objects

## Tracing & Spans

Stack Auth includes built-in tracing with `startSpan()`. Spans measure timed operations and automatically link to events, session replays, and other spans.

### Client-side span

```ts
await app.startSpan("checkout.click", async (span) => {
  app.trackEvent("checkout.started", { source: "button" }); // auto-linked to span
  await submitOrder();
  span.setStatus("ok");
});
```

The callback form auto-ends the span when the callback returns. If it throws, the span records the error and re-throws.

### Server-side span

```ts
await serverApp.startSpan("api.create-order", req, async (span) => {
  const body = await req.json();
  span.setAttribute("item_count", body.items.length);
  const order = await createOrder(body);
  span.setStatus("ok");
  return order;
});
```

Pass the request as the 2nd argument (same as `trackEvent`) to extract user identity and trace context.

### Distributed tracing (client to server)

When a `fetch` is made inside an active span, the SDK automatically injects an `x-stack-trace` header on same-origin requests. Server-side `startSpan()` reads this header and makes the server span a child of the client span:

```ts
// Client — trace context auto-injected into fetch
await app.startSpan("checkout.flow", async (span) => {
  app.trackEvent("checkout.started", { total: 99.99 });
  const res = await fetch("/api/checkout", { method: "POST", body: JSON.stringify({ items }) });
  span.setStatus(res.ok ? "ok" : "error");
});

// Server — automatically becomes a child span
await serverApp.startSpan("api.checkout", req, async (span) => {
  const order = await processOrder(await req.json());
  span.setStatus("ok");
  return order;
});
```

### Nested spans

Nested `startSpan()` calls automatically detect the parent via `AsyncLocalStorage`, which is available in Node.js, Deno, Bun, Cloudflare Workers, and all serverless runtimes:

```ts
await serverApp.startSpan("order.process", async (parent) => {
  // Child spans automatically become children of "order.process"
  await serverApp.startSpan("order.validate", async (v) => {
    await validateOrder(order);
    v.setStatus("ok");
  });
  await serverApp.startSpan("order.charge", async (c) => {
    await chargeCard(order.paymentMethod);
    c.setStatus("ok");
  });
  parent.setStatus("ok");
});
```

This also works on the client when `AsyncLocalStorage` is available (e.g., Next.js SSR). In pure browser environments, context is lost across `await` — use the span callback argument and pass `parentSpanId` explicitly for nesting:

```ts
await app.startSpan("outer", async (outerSpan) => {
  await someAsyncWork();
  // Pass the parent span object directly — traceId and parentId are inherited
  await app.startSpan("inner", { parent: outerSpan }, async (innerSpan) => {
    innerSpan.setStatus("ok");
  });
  outerSpan.setStatus("ok");
});
```

| Environment | Auto-parent across `await` | Fallback |
| --- | --- | --- |
| Node.js, Deno, Bun | Yes | N/A |
| Serverless (Vercel, Lambda, Cloudflare Workers) | Yes | N/A |
| Browser (SSR with AsyncLocalStorage) | Yes | N/A |
| Browser (client-side JS) | No | Pass `parentSpanId` explicitly |

### Manual spans

For cases where the callback form doesn't fit (e.g., event emitters), use the manual form:

```ts
const span = serverApp.startSpan("file-upload");
try {
  await uploadFile(file);
  span.setStatus("ok");
} catch (err) {
  span.recordException(err);
  throw err;
} finally {
  span.end();
}
```

### Span API

| Method | Description |
| --- | --- |
| `span.spanId` | UUID of this span |
| `span.traceId` | UUID of the trace this span belongs to |
| `span.isEnded` | Whether `end()` has been called |
| `span.setAttribute(key, value)` | Set a key-value attribute |
| `span.setAttributes(attrs)` | Bulk-set attributes |
| `span.setStatus(status, message?)` | Set status: `'ok'` \| `'error'` \| `'unset'` |
| `span.recordException(error)` | Set status to `'error'` and attach error metadata |
| `span.end()` | End the span (idempotent) |

## Super Properties

Register properties that are automatically merged into every event's `data`:

```ts
app.register({ plan: "pro", environment: "production" });

app.trackEvent("checkout", { amount: 99 });
// → data: { plan: "pro", environment: "production", amount: 99 }

app.unregister("plan"); // remove a single key
```

Event-specific data takes precedence over super properties. Works on both client and server apps.

## Sampling

Control what percentage of traces are kept with `tracesSampleRate`:

```ts
const app = new StackClientApp({
  projectId: "...",
  analytics: {
    tracesSampleRate: 0.1, // keep 10% of traces
  },
});
```

- Only **root spans** are sampled — child spans inherit the parent's decision
- Sampled-out spans return a no-op: the callback still runs, but nothing is sent
- Events (`trackEvent`, `captureException`, auto-captured) are **never** sampled
